# ADR 0005: API バージョニング (`/api/v1`) を採用しない

## ステータス

Accepted（2026-05-01）

## コンテキスト

REST API のパス設計において、`/api/v1/videos` のようなバージョンプレフィックスを置くべきかを検討した。
Phase 1〜2 の最初の実装では `/api/v1/` を一旦採用していたが、本リポジトリ内の他プロジェクト（Slack）が
**bare path**（`/channels`, `/me`）で書かれており不整合だったこと、および学習用途・単一クライアント前提で
`v1` を切る合理性が薄いことから、Phase 2 終盤で見直しを行った。

検討材料：

- 利用者は **同リポジトリの Frontend (Next.js) だけ**。同時デプロイのため互換性の段階移行が要らない
- 学習目的は「動画変換パイプライン / 状態機械 / レコメンド責務分離」であり、API ガバナンスは対象外
- 本物の API バージョニングには **deprecation policy / Sunset ヘッダ / バージョン併走運用** がセットになる。半端な `v1` プレフィックスだけ付けても飾り
- Slack プロジェクト側は bare path で運用しており、リポジトリ内で揃えたい

## 決定

**API パスにバージョンプレフィックスを付けない**を採用する。

- ルート定義は `resources :videos` のように bare で書く
- 形のうえで API かそうでないかの区別が必要なケース（例: 認証必須スコープ）は、ルーティングではなく
  **コントローラ側のフィルタ**または **ScopeUI** 単位の名前空間で表現する
- 将来 `/v2` を切る必要が生じた時点で別 ADR を起こし、その時に決める

## 検討した選択肢

### 1. プレフィックスなし (`/videos`) ← 採用

- 利点: Slack プロジェクトと記法が揃う
- 利点: ルート定義 / フロント / テストすべてが短くなり、ノイズが減る
- 欠点: 本番想定で「API はちゃんと `/api/` を切るのが普通」という直感に逆行する印象

### 2. `/api` のみ (バージョンなし)

- 利点: SSR ルート (Next.js 側) と URL 衝突した時の名前空間として有用
- 欠点: 本プロジェクトでは Rails が API mode で **動画ファイル等の SSR ルートは持たない**。名前空間衝突は起きない
- 欠点: それでも 1 階層深くなるノイズは残る

### 3. `/api/v1` (元の実装)

- 利点: 本番化時に違和感なく拡張できる
- 利点: REST API の業界慣習と一致
- 欠点: バージョン管理の規律（deprecation / 併走）と切り離されると形だけになる
- 欠点: 単一クライアント・同時デプロイの本プロジェクトでは互換維持の必要性がない

## 採用理由

- **学習目的との整合**: 本プロジェクトの技術課題は API ガバナンスではない。読むコード量を減らし、状態機械やキューの方に脳を使えるようにしたい
- **YAGNI**: 互換性の段階移行が必要な瞬間に切ればよい。先取りするコストが学習効果を上回らない
- **既存プロジェクトとの一貫性**: Slack は bare path。ライブラリ的な扱いではなく "lab" として横断的に読まれることを意識し、記法を揃える
- **可逆性**: 後から `/api/v1` に移すのは routes の `scope` で囲うだけで済む。意思決定を遅らせるコストが低い

## 却下理由

- `/api/v1`: 単一クライアントで互換性運用がない以上、形だけのプレフィックスになる
- `/api` のみ: Rails が API mode で SSR ルートを持たないため、名前空間として要らない

## 引き受けるトレードオフ

- **本番想定との乖離**: 実プロダクトでは `/api/v1` が一般的。本番化用 Terraform / 設計図には ALB のパスベースルーティングで `/api` を切る前提を残し、コードと別レイヤーで吸収する
- **将来の切替コスト**: `/v2` を切る判断が出たときは routes と Frontend の両方を一斉に直す必要がある。ただし Phase 5 までに `/v2` が必要になるシナリオは現実的でない
- **求人などで API 設計の素養を問われた際の説明責任**: バージョニングの考え方は理解した上で「この規模で要らない」と判断したことを ADR で示す（この文書がそれ）

## このADRを守るテスト / 実装ポインタ

- `youtube/backend/config/routes.rb` — `scope` を使わず bare で書く
- `youtube/backend/test/integration/videos_api_test.rb` — `/videos` をテスト
- `youtube/frontend/src/lib/videos.ts` — 同じく bare path で fetch
