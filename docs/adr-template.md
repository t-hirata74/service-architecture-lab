# ADR テンプレート

新しい設計判断を残す時は、このテンプレートを `<service>/docs/adr/NNNN-<決定の見出し>.md` にコピーして埋める。

- ファイル名の `NNNN` は 4 桁ゼロ埋め（`0001`, `0002`, ...）
- 1 ファイル **50〜150 行** が目安。長文ドキュメントにしない
- 既存決定を覆す場合は、既存 ADR を**削除せず** Status を `Superseded by ADR NNNN` に変更し、新しい ADR を起こす
- プロジェクトあたり **最低 3 本**

---

## テンプレート本体（ここから下をコピー）

```markdown
# ADR NNNN: <決定の見出し>

## ステータス

Accepted（YYYY-MM-DD）
<!-- Proposed / Accepted / Superseded by ADR NNNN のいずれか -->

## コンテキスト

<!-- 何を解こうとしているか / 制約 / スコープ。3〜5 段落 -->
<!-- 「将来スケールするとどうか」「ローカル完結という制約」など本リポジトリ固有の制約も書く -->

## 決定

**<採用したアプローチを 1 行で>** を採用する。

- 構成要素 1
- 構成要素 2
- 構成要素 3

## 検討した選択肢

### 1. <採用案> ← 採用

- 利点
- 利点

### 2. <代替案 A>

- 利点
- 欠点

### 3. <代替案 B>

- 利点
- 欠点

<!-- 比較が表で書きやすい時は表でも OK -->

## 採用理由

- **学習価値**：このプロジェクトの目的（"技術課題の再現"）に対して何が学べるか
- **アーキテクチャ妥当性**：実プロダクトでも採用される設計か
- **責務分離**：他レイヤーを汚さないか
- **将来の拡張性**：スケールさせる時にどう発展できるか

## 却下理由

- 案 A: なぜ取らなかったか（1〜2 行）
- 案 B: なぜ取らなかったか

## 引き受けるトレードオフ

- **<運用コスト>**：何が増えるか / なぜ許容するか
- **<整合性 / 順序保証など>**：何を諦めたか / どこで吸収するか
- **<将来の課題>**：別 ADR で扱う予定の論点

## このADRを守るテスト / 実装ポインタ

<!-- 任意：ADR が宣言した不変条件を守っているテストや実装位置 -->
- `slack/backend/test/integration/foo_test.rb` — XX を検証
- `slack/backend/app/models/bar.rb:42` — 単調増加ガード

## 関連 ADR

- ADR NNNN: <関連する別の決定>
- ADR NNNN（予定）: <派生して立てる予定の論点>
```

---

## 書く時のコツ

- **「なぜ採用したか」より「なぜ他を取らなかったか」を厚く書く**：ADR の価値は却下理由に出る
- **トレードオフを正直に書く**：「諦めた点」が無い ADR は信用されない
- **コードからは読み取れないことを書く**：「Redis を使う」だけならコードを読めば分かる。「他に async / Socket.IO / SSE を検討した」がコードでは分からない部分
- **未来の読者向け**：半年後の自分 / 採用担当者 / 他開発者が読めば設計意図が伝わる文体にする
- **長くしない**：迷ったら削る。150 行を超え始めたら「別 ADR に分離できないか」を考える

---

## ADR 番号の運用

- 番号はプロジェクトごとに独立（`slack/docs/adr/0001`, `youtube/docs/adr/0001` でよい）
- 採番は連番。欠番が出ても詰めない
- ステータス変更（Accepted → Superseded）も commit する。差分が履歴に残る形を保つ
