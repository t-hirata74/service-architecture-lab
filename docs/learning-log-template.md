# 学びログテンプレート

各サービスの `architecture.md` の末尾、または独立した `learning-log.md` に追加する「学びログ」セクションの雛形。

**目的**: 成功例だけでなく **失敗・妥協・次の改善案** を残すことで、設計プロセスの解像度を上げる。読み手にとっては「思考の軌跡」が伝わり、書き手にとっては次のサービスで同じ罠を踏まない土台になる。

---

## 配置の原則

- **`architecture.md` の末尾セクション**として置く（独立ファイル化は ADR 数が増えて冗長になった場合のみ）
- 完成（MVP / Phase 5 など）後に書き始め、レビューやリファクタの度に追記する
- 共通化された知見は `docs/coding-rules/` や `docs/operating-patterns.md` に昇華して、本ログにはサービス固有の話だけ残す

---

## セクション構成（雛形）

`architecture.md` の末尾に以下をコピーして埋める。すべて埋める必要はないが、**「採用しなかった選択肢」と「次に作るなら変える点」は最低限書く**ことを推奨。

```markdown
## 学びログ

### 採用しなかった選択肢と理由

ADR 本文に書かない/書ききれない、より細かい選択肢の捨て理由を残す。

- **{選択肢 A}**: {捨てた理由 — 例: ローカル完結方針と矛盾する / 学習対象から外れる / 実装コストに対し学びが薄い}
- **{選択肢 B}**: {捨てた理由}

### 踏んだ落とし穴

実装中に詰まった具体的な罠。**横断的な落とし穴は `docs/coding-rules/` に移し、本セクションには本サービス固有のものだけ残す。**

- **{現象}** — 原因: {何だったか} / 対処: {どう直したか} / 学び: {次に同じものを書くときのチェック}
- {可能ならコミットや PR へのリンクを足す}

### もう一度作るなら変える点

完成後に「最初からこうしておけば」と思った点。コード変更を伴わなくてよい。

- **{変える点}**: {理由 — 例: スキーマを最初から N:M で切れば後段の denormalize が要らなかった}
- **{変える点}**: {理由}

### 残した宿題（意図的に捨てたスコープ）

「完成の定義」に従い、技術課題の理解と無関係なので意図的に捨てた機能。**「単に時間切れ」と「意図的に捨てた」は分けて書く**ことが重要（前者は将来の reminder、後者はスコープ判断の説明）。

- **意図的に捨てた**: {何を / なぜ}
- **時間切れ / 次フェーズ**: {何を / 何があれば再開するか}

### 横断ドキュメントに昇華した知見

このサービスで踏んで `docs/` 配下に昇華したパターン。リンクで参照する。

- [`docs/coding-rules/{lang}.md`](../../docs/coding-rules/{lang}.md) — {見出し名}
- [`docs/operating-patterns.md`](../../docs/operating-patterns.md) — {見出し名}
```

---

## 書き方のコツ

- **WHAT より WHY と WHAT NEXT に紙幅を割く**: 何が起きたかは git log に残るが、なぜ / 次どうするかは文書でしか残らない
- **横断知見は本ログに置かず昇華する**: `docs/coding-rules/` や `docs/operating-patterns.md` に移して、本ログにはサービス固有の話だけ残す
- **失敗を矮小化しない**: 「これくらいは…」と削るとログの意味が消える。トレードオフを正直に書く
- **ADR と重複させない**: ADR は「採用した結論」が主役。本ログは「ADR に書ききれない周辺の判断・反省・宿題」が主役。両者は補完関係
- **更新時刻を入れない**: ファイル全体の更新は git log で追える。日付を散らすと逆に陳腐化が早い

## 例（参照用の架空ケース）

> 以下は雛形イメージのための架空例。実プロジェクトの内容ではない。

```markdown
## 学びログ

### 採用しなかった選択肢と理由

- **Redis Streams での fan-out**: ローカル完結方針には合うが、Rails 標準の ActionCable + Redis Pub/Sub で十分シンプルに書けるため採用せず。学習価値は重複
- **gRPC で Rails ↔ ai-worker 接続**: 本リポは「内部 ingress = REST + 共有トークン」で統一しており、サービスごとに通信プロトコルを変えると対比軸がブレる

### 踏んだ落とし穴

- **`select_related` 漏れで N+1 が再発** — 原因: ビュー追加時にテストが N+1 件数を fixate していなかった / 対処: `assertNumQueries` を全 list view に追加 / 学び: 「N+1 が起きないこと」をテストで不変条件化する。`docs/testing-strategy.md` 「N+1 不変条件試験 (Django)」に昇華済み

### もう一度作るなら変える点

- フォローグラフを `Follow(follower_id, followee_id)` ではなく `Edge(src_user_id, dst_user_id, kind)` にして、後の「ブロック」「ミュート」も同テーブルで扱えるようにしておきたかった

### 残した宿題

- **意図的に捨てた**: ストーリーズ / DM — タイムライン生成戦略の学習対象外
- **時間切れ**: タイムラインのキャッシュ無効化戦略 — 現状は毎リクエスト DB 直当て。次フェーズで `recount` management command と組み合わせて検討

### 横断ドキュメントに昇華した知見

- [`docs/coding-rules/python.md`](../../docs/coding-rules/python.md) — Django ORM `select_related` 明示 / `assertNumQueries` 不変条件試験
- [`docs/operating-patterns.md`](../../docs/operating-patterns.md) — fan-out on write + soft delete
```
