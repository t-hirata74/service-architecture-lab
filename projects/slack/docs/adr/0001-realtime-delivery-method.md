# ADR 0001: リアルタイムメッセージ配信方式の選定

## ステータス

Accepted（2026-04-30）

## コンテキスト

Slack 風チャットを実装するにあたり、**サーバーから複数クライアントへメッセージを低遅延で push する仕組み**が必要。

このプロジェクトのスコープ：

- ローカル完結（外部サービス禁止）
- 同時接続規模はローカル検証なので最大 100 接続程度を想定
- ただし「将来スケールした場合の設計妥当性」も学習対象に含む
- 双方向通信（クライアント → サーバーのメッセージ送信、サーバー → クライアントの配信、入力中インジケーター等）が必要

## 決定

**Rails ActionCable + Redis Pub/Sub** を採用する。

- Rails ActionCable をリアルタイム購読層として使用
- 複数 Rails プロセス間のメッセージ伝播は Redis Pub/Sub アダプタで行う
- クライアントは ActionCable 公式 JS ライブラリ経由で WebSocket 接続

## 検討した選択肢

### 1. ActionCable + Redis Pub/Sub アダプタ ← 採用

- Rails 標準でフルセットを得られる
- Redis を介すことで Rails プロセスの水平スケールが可能
- 「実プロダクトでも採用される構成」を学習できる

### 2. ActionCable + async（インメモリ）アダプタ

- 開発が最も簡単
- ただし単一プロセス前提なので、スケール議論ができない
- 学習目的（"将来スケールするならどう設計するか"）と合わない

### 3. Node.js + Socket.IO を独立サーバーとして併設

- リアルタイム機能は Node に分離する構成
- 業界でよく見るが、本リポジトリの「Rails が backend を担う」方針と合わない
- ランタイムが3つ（Rails / Python / Node）になり責務分割が散らかる

### 4. Server-Sent Events（SSE）

- HTTP ベースで実装が単純
- ただし**双方向ではない**ため、クライアント → サーバー送信は別途 REST 必要
- 入力中インジケーター等のクライアント発のイベント送信が分散して扱いづらい

### 5. ロングポーリング

- WebSocket 不要で実装可能
- リアルタイム性に劣る、コネクション数増大時のスケールが苦しい
- 「Slack の技術課題を再現する」という目的に対して妥当性が低い

## 採用理由

- **学習価値**：ActionCable は Rails の中で最も独特な層の一つで、購読/ブロードキャスト/認証の流れを手で書ける学習価値が大きい
- **アーキテクチャ妥当性**：Redis Pub/Sub による fan-out は Slack 自身も初期に採用していた設計に近く、本プロジェクトの「技術課題の再現」目的に合致
- **責務分離が綺麗**：双方向通信が ActionCable に集約されるため、Rails 内の他レイヤー（CRUD API など）が肥大化しない
- **将来の拡張性が説明できる**：Redis Pub/Sub → 必要なら Kafka/NATS 等への置換、Sticky session、ActionCable サーバー分離、などの議論につながる

## 却下理由

- async アダプタ：スケール議論ができないため学習目的に合わない
- Socket.IO 併設：ランタイム数が増え、本リポジトリの3層構成思想と衝突
- SSE：双方向性が必要な要件に不適合
- ロングポーリング：Slack の技術課題（リアルタイム fan-out）を再現できない

## 引き受けるトレードオフ

- **Redis を docker-compose に常駐させる運用コスト**：開発環境のリソース消費が増える。許容。
- **Pub/Sub のメッセージ配信保証は at-most-once**：Rails プロセスがダウンしている購読者にはメッセージが届かない。永続化は DB に書いた上で WebSocket 経由は通知扱いとする設計で吸収する。
- **メッセージ順序保証は弱い**：単一チャンネル内では DB の `created_at` を信頼し、クライアント側で並び替える方針を取る。順序を厳密に保証したい場合は別途キューを介す設計が必要。
- **WebSocket 接続のスケール問題**：1プロセスあたりの接続数上限（OS のファイルディスクリプタ等）の議論は将来 ADR を起こす。
