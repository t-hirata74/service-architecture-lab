# ADR 0003: データベースに MySQL を採用する

## ステータス

Accepted（2026-04-30）

## コンテキスト

本プロジェクトは Slack のアーキテクチャ再現を目的とする。Backend は Rails、データ永続化が必要な領域はチャット系の典型的なリレーショナルデータ（users / channels / memberships / messages）が中心。

DB の選択は以下の観点で意思決定する必要がある：

- **対象サービスの実設計との一致**（学習目的・「再現する」方針に直結）
- **Rails との相性**
- **将来のスケール議論ができる構成**（学習対象として）
- **開発者の慣れ・運用知識の蓄積**

## 決定

**MySQL 8.x** を採用する。docker-compose では `mysql:8.0` 公式イメージを使用する。

## 検討した選択肢

### 1. MySQL 8.x ← 採用

- **Slack 自身が MySQL を採用**（公開されたエンジニアリングブログ複数で言及。後に Vitess でシャード化）
- **GitHub も MySQL + Vitess** で、本リポジトリの将来的な GitHub プロジェクトでも一貫性が取れる
- Rails は MySQL を一級でサポート（adapter は `mysql2` gem）
- 開発者本人の慣れがあり、運用知識が蓄積している

### 2. PostgreSQL 16

- Rails コミュニティの近年の推奨はこちら寄り
- JSONB / 配列型 / 全文検索 (`tsvector`) などのリッチな型が強み
- ただし **Slack の実設計と一致しない** ため、学習目的（"サービスが解いている技術課題を再現する"）から外れる
- 全文検索は ai-worker や別ストア（後述）に委ねる方針なら、PostgreSQL の優位はそこまで効かない

### 3. SQLite

- ローカル完結方針には最も親和的
- ただし複数プロセスからの同時書き込み・レプリケーション議論ができないため、**学習対象としての将来スケール議論に乗らない**
- 不採用

## 採用理由

- **Slack の実設計と一致する**：これが最大の理由。本プロジェクトは「サービスが解いている技術課題を再現する」を掲げており、DB 選定もその思想に従うべき
- **将来のシャーディング議論につながる**：MySQL → Vitess は Slack / GitHub / YouTube が辿ったルート。学習対象として ADR で言及しやすい
- **Rails ↔ MySQL は実プロダクト運用例が膨大**：チューニング・監視のリソースが豊富で、学習価値が高い
- **個人の運用知識が活きる**：スキーマ設計・チューニング・トラブルシュートで詰まりにくい

## 却下理由

- **PostgreSQL**：技術的には十分強力だが、Slack 再現という文脈では弱く、本プロジェクトでは旨味が薄い
- **SQLite**：将来スケール議論ができない（学習対象として致命的）

## 引き受けるトレードオフ

- **JSONB は使えない**：MySQL の `JSON` 型は機能的に劣る。リッチな構造データは TEXT + アプリ層パース、または別ストア（Redis 等）に逃がす
- **配列型がない**：Postgres の `text[]` のような型は使えない。中間テーブルで正規化する
- **全文検索が弱い**：MySQL の `FULLTEXT` インデックスは日本語処理が貧弱。検索の本格実装は ai-worker 側で別途設計する（ADR 0xxx で扱う）
- **timestamptz がない**：タイムゾーンは Rails アプリ側で UTC 統一する運用にする（Rails のデフォルト挙動と整合）

## 他プロジェクトへの波及

- **YouTube プロジェクト**：YouTube 自体は Bigtable / Spanner だが、ローカル学習用としては MySQL で問題なく、本リポジトリ内の一貫性も保てる
- **GitHub プロジェクト**：GitHub も MySQL + Vitess。むしろ MySQL のほうが正確
- ただしこの ADR は **slack プロジェクトに限定** した決定。他プロジェクトは各々の ADR で改めて判断する
